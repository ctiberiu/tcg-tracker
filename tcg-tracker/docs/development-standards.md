## 1. Architecture Overview

**Pattern:** Two-runtime monolith sharing a Supabase backend

```
┌─────────────────┐    ┌──────────────────┐
│  React SPA      │    │  Node.js Scraper  │
│  (Vercel)       │    │  (GitHub Actions) │
└────────┬────────┘    └────────┬─────────┘
         │                      │
         └──────────┬───────────┘
                    │
          ┌─────────▼──────────┐
          │  Supabase          │
          │  (PostgreSQL + Auth│
          │   + Edge Functions)│
          └────────────────────┘
```

**Stack:**
- Frontend: React 19 + TypeScript ~5.9 + Vite 8 + Tailwind CSS 4
- Scraper: JavaScript (ESM) + Playwright + stealth plugin
- Database: Supabase (PostgreSQL with RLS)
- Deployment: Vercel (frontend), GitHub Actions (scraper), Supabase Edge Functions (trigger)

---

## 2. Project Principles

- **KISS** — avoid over-engineering; this is a single-user tool
- **Separation of concerns** — frontend reads data, scraper writes data, edge function bridges them
- **Fail gracefully** — scraper should continue to next store on individual store failure
- **No unnecessary abstractions** — direct Supabase client calls, no ORM or repository pattern
- **Type safety** — TypeScript in frontend; shared types in `src/lib/types.ts`

## 3. Layer Responsibilities

### Frontend (`src/`)
- **Pages** (`src/pages/`) — route-level components; contain page layout and state
- **Hooks** (`src/hooks/`) — encapsulate all Supabase data access and state management
- **Components** (`src/components/`) — reusable UI elements; no direct data fetching
- **Lib** (`src/lib/`) — shared utilities, client initialization, type definitions

**Rules:**
- Components must not call Supabase directly — use hooks
- Pages may use multiple hooks
- Types shared between frontend modules go in `src/lib/types.ts`

### Scraper (`scraper/`)
- Single-file architecture (`scraper.js`)
- Store-specific scrape functions receive a Playwright `page` and `store` config
- All DB operations use service role Supabase client

### Database (`supabase/migrations/`)
- Sequential numbered migration files
- RLS enabled on all tables
- Triggers and functions defined in migration files

## 4. Data Contracts

### Product (DB → Frontend)
| Field        | Type                  | Nullable | Notes                        |
|--------------|-----------------------|----------|------------------------------|
| `id`         | UUID                  | No       | Generated by DB              |
| `store_name` | string                | No       |                              |
| `store_id`   | UUID                  | Yes      | FK to stores                 |
| `title`      | string                | No       |                              |
| `price`      | number                | Yes      | In RON                       |
| `url`        | string                | No       | Unique; used for upsert      |
| `image_url`  | string                | Yes      |                              |
| `in_stock`   | boolean               | No       |                              |
| `is_notified`| boolean               | No       | Default false                |
| `first_seen` | ISO 8601 timestamp    | No       | Preserved across upserts     |

### Scraper Output Format
Each scrape function must return `Array<{ title, price, url, image_url, store_name, store_id, in_stock }>`.

### Upsert Strategy
- Conflict key: `url`
- On conflict: update `price`, `image_url`, `in_stock`
- `first_seen` preserved via database trigger

## 5. Error Handling

### Frontend
- Hooks expose `error: string | null` state for UI display
- Use try/catch in async operations; surface errors via state
- No global error boundary (not yet implemented)

### Scraper
- Per-store try/catch: individual store failures log and continue
- Top-level try/catch: marks scrape_run as failed in DB
- Process exits with code 1 on critical failure

### Edge Function
- Return structured JSON errors with appropriate HTTP status codes
- Always include CORS headers in error responses

---

## 6. Logging Standards

### Scraper
- Use `console.log()` for informational messages (product counts, sync results)
- Use `console.error()` for errors and failures
- Prefix store-specific messages with `  ${store.name}:`
- Log summary counts after scraping and syncing

### Frontend
- No structured logging; use browser console for development
- Do not log sensitive data (tokens, passwords)

## 7. Configuration Management

### Hierarchy
1. Environment variables (`.env` files, Vercel env, GitHub secrets, Supabase secrets)
2. No config files beyond env vars

### Naming Conventions
- Frontend env vars: `VITE_` prefix (exposed to client)
- Scraper env vars: `SUPABASE_URL`, `SUPABASE_KEY`, `RESEND_API_KEY`, `ALERT_EMAIL_TO`
- Edge function: Deno `Deno.env.get()`

### Sensitive Data
- Never commit `.env` with real values
- Use `.env.example` files as templates
- Service role keys only in server-side contexts (scraper, edge functions)

## 9. Security & Compliance

### Authentication
- Supabase Auth with email/password
- Single-user enforcement via `VITE_ALLOWED_EMAIL` env var
- Session managed by Supabase client; checked in `useAuth` hook and `ProtectedRoute`

### Secure Coding
- HTML escaping in email templates (`escapeHtml()` in scraper)
- URL sanitization for email links (`sanitizeUrl()` — allows only http/https)
- Supabase parameterized queries (no raw SQL in frontend)
- Input length limits on login form fields

### Data Protection
- No PII stored beyond the admin user's email in Supabase Auth
- Public product data only (prices, URLs, titles from public store pages)

### Dependency Management
- Pin major versions in `package.json`
- Use `npm ci` in CI for deterministic installs
- Playwright browsers installed fresh in each CI run

## 10. Directory Layout

```
tcg-tracker/
├── src/
│   ├── pages/          # Route-level components
│   ├── hooks/          # Data hooks (useAuth, useProducts, useStores)
│   ├── components/     # Shared UI (AppSidebar, ProtectedRoute)
│   ├── lib/            # Supabase client, types
│   ├── assets/         # Images
│   ├── App.tsx         # Router setup
│   ├── main.tsx        # Entry point
│   └── index.css       # Tailwind + theme
├── scraper/
│   ├── scraper.js      # Main scraper script
│   ├── run-migrations.js
│   └── package.json
├── supabase/
│   ├── migrations/     # Numbered SQL migrations
│   └── functions/      # Edge Functions (Deno)
├── .github/workflows/  # GitHub Actions
├── docs/               # Project documentation
└── [config files]      # vite.config.ts, tsconfig.json, etc.
```

### File Naming
- React components: PascalCase (e.g., `DashboardPage.tsx`)
- Hooks: camelCase with `use` prefix (e.g., `useProducts.ts`)
- Lib files: camelCase (e.g., `supabase.ts`, `types.ts`)
- Migrations: numbered prefix (e.g., `001_create_products.sql`)

### Where to Place New Features
- New pages: `src/pages/`
- New shared components: `src/components/`
- New data hooks: `src/hooks/`
- New types: `src/lib/types.ts`
- New scraper functions: `scraper/scraper.js` (add to `SCRAPER_MAP`)
- New DB tables: `supabase/migrations/<next_number>_<name>.sql`

## 11. Design Principles

### Coding Style
- Functional React components only (no class components)
- Custom hooks for all data access
- Tailwind CSS utilities for styling (no inline styles or CSS modules)
- TypeScript strict mode for frontend

### Naming Conventions
- **Components/Pages**: PascalCase (`DashboardPage`, `AppSidebar`)
- **Hooks**: `use` prefix, camelCase (`useProducts`, `useAuth`)
- **Database columns**: snake_case (`store_name`, `first_seen`)
- **TypeScript interfaces**: PascalCase (`Product`, `Store`, `ScrapeRun`)
- **CSS**: Tailwind utility classes; custom theme tokens in `index.css`

### Code Validation (before review)
Run these commands before submitting changes:

| Stack     | Build                    | Lint              |
|-----------|--------------------------|-------------------|
| Frontend  | `npm run build`          | `npm run lint`    |

Both must pass cleanly. Fix lint issues with manual corrections (no `--fix` auto-fix for ESLint in this project).

## 12. Failure Handling & Resilience

### Scraper
- **Per-store isolation**: each store scraped in its own browser context; failure in one does not affect others
- **Timeout**: 30s for page navigation, 15s for selector waits
- **No retry**: failed stores are skipped (logged and moved on)
- **Scrape run tracking**: status recorded in `scrape_runs` table for admin visibility

### Frontend
- **Polling timeout**: Admin scrape polling stops after 5 minutes
- **No offline support**: requires active network connection
- **Graceful degradation**: email alerts are optional (skip if API key not configured)

### Edge Function
- **CORS**: validates origin against allowlist
- **Auth**: verifies Supabase JWT before processing
- **No retry**: single attempt to dispatch GitHub Actions workflow
